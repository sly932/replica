# MaaS 平台系统架构

> 本文档描述统一大模型推理服务（MaaS, Model-as-a-Service）的整体架构、核心模块、数据流与多租户隔离设计。MaaS 是基建与中台事业部对内提供的统一推理底座，三条产品线（灵犀对话模型、悟空代码模型、磐石取数模型）的线上推理流量全部经由 MaaS 进出。平台负责人赵启明（zhaoqiming01）。推理网关实现细节找推理后端简如，GPU 调度与显存问题找 SRE 乔宁。

## 一、整体定位

MaaS 屏蔽底层 GPU 集群、模型权重、推理引擎（vLLM / TensorRT-LLM / SGLang）的差异，对业务方暴露统一的 OpenAI 兼容接口（`/v1/chat/completions`、`/v1/completions`、`/v1/embeddings`）。业务方只感知「模型名 + API Key + 配额」，不感知物理部署。一句话：**业务组要扩模型、加配额，都得走 MaaS，找赵启明审批**（流程见《模型接入与配额申请 SOP》，slug: model-quota-sop）。

## 二、四大核心模块

### 1. 推理网关（Inference Gateway）
- 统一入口，承载鉴权（API Key + 租户 ID）、限流（QPS/TPM 双维度令牌桶）、请求改写、协议适配。
- 部署形态：无状态多副本，前置 L7 负载均衡，按租户做一致性哈希以利于 KV-Cache 复用。
- 网关层维护「模型名 → 后端服务组」的路由表，热更新，变更走配置中心灰度。
- 细节实现（鉴权链路、令牌桶参数、改写规则）由简如维护。

### 2. 模型托管（Model Serving）
- 管理模型权重的加载、版本、副本数、推理引擎选型。
- 每个模型以「模型服务组」为单位部署，组内 N 个推理实例（pod），每实例绑定 1–8 张 GPU 不等。
- 支持的模型类型：**对话类（Chat，含多轮 / 工具调用）、补全类（Code Completion / FIM）、Embedding 类、Reranker 类**。多模态（图文）目前仅灰度，未正式对外。
- 模型版本以 `model_name@version` 标识，支持灰度（按租户百分比）与回滚。

### 3. 配额与计费（Quota & Billing）
- 配额三个维度：**QPS（每秒请求数）、TPM（每分钟 token 数，含 prompt+completion）、GPU 配额（预留显卡数，用于独占部署场景）**。
- 计费按 token 量分模型档位计价，月度结算到各产品线成本中心。计费数据 T+1 落库。
- 配额超限触发限流（429），处置见《MaaS 值班与限流预案》（slug: oncall-throttle）。

### 4. 模型路由（Model Router）
- 同一逻辑模型名后可挂多个物理后端（不同引擎 / 不同卡型 / 不同区域），路由器按权重、健康度、负载做选择。
- 支持**降级路由**：主模型不可用或超时时，按预设降级链路由到备用模型（如大参数模型降级到蒸馏小模型）。降级策略由各产品线与 MaaS 共同约定。

## 三、核心数据流

```
业务方 SDK
  → 推理网关（鉴权 / 限流 / 协议适配）
  → 模型路由（选后端：权重 + 健康度 + 负载）
  → 模型托管（推理实例 vLLM/TRT-LLM 执行）
  → 流式 / 非流式返回
  ↘ 异步：用量埋点 → 配额计费（T+1 结算）
```

旁路上报：每次请求的 token 用量、时延、首 token 时延（TTFT）、状态码异步写入观测链路，供配额核销与 SLA 统计。

## 四、多租户隔离

- **逻辑隔离（默认）**：所有租户共享模型服务组，靠网关层 QPS/TPM 配额 + 优先级队列做软隔离。绝大多数业务走这一档。
- **物理隔离（独占）**：对延迟敏感或合规要求高的租户，可申请 GPU 配额做独占部署，模型实例不与他人共享。成本计入该产品线。
- 租户优先级分 P0/P1/P2 三档，资源紧张时按优先级抢占；**具体抢占阈值与各档保障比例由 SRE 侧动态调整，未在本文固化**。

## 五、SLA 概要

- 推理网关可用性目标 **99.9%**（月度）。
- 在线对话类 P99 首 token 时延目标 **< 1.2s**（标准档），整体 P99 完成时延视输出长度而定。
- 配额变更生效时效、紧急扩容时效见 SOP 文档；常规变更 T+1，**紧急 / 超大配额特批不在常规时效内，需另行特批**。
