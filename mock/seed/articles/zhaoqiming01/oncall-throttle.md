# MaaS 值班与限流预案 + 踩坑

> 本文档供 MaaS 值班同学使用，覆盖推理超时、配额超限、降级路由的处置预案，以及历史踩坑记录。值班轮班：赵启明 / 简如 / 乔宁三人轮值，告警进飞书「MaaS-Oncall」群。涉及业务影响判断、跨组协调升级找赵启明（zhaoqiming01）。

## 一、值班职责与升级路径

- **P0（核心模型整体不可用 / 网关挂）**：5 分钟内响应，拉群同步，30 分钟内给缓解措施。升级到赵启明。
- **P1（单产品线受影响 / 部分降级）**：15 分钟内响应。
- **P2（个别租户报错 / 配额咨询）**：当班处理，工作时间内回复。

## 二、常见故障处置预案

### 1. 推理超时（请求 5xx / 超 timeout）
- **现象**：网关返回 504，或 SDK 侧报 `Read timeout`；TTFT 飙高。
- **排查顺序**：先看模型服务组负载（QPS/排队深度）→ 看 GPU 利用率与显存 → 看是否有长 prompt 拖慢批处理。
- **处置**：
  - 队列积压 → 临时提副本数（需乔宁确认有卡）；
  - 个别实例卡死 → 摘流 + 重启该 pod；
  - 整组退化 → 触发**降级路由**（见下）。

### 2. 配额超限（429）
- **现象**：业务方收到 `429 Too Many Requests`，响应头带 `X-RateLimit-Remaining: 0`。
- **判断**：是撞 QPS 还是撞 TPM——看网关限流维度标签。长输出场景多为撞 TPM。
- **处置**：
  - 短时突发：建议业务方退避重试（指数退避，初始 1s）。
  - 持续超限：让业务方走《模型接入与配额申请 SOP》（slug: model-quota-sop）正式扩容；**值班不直接私自改配额**，常规扩容 T+1 生效，紧急情况升级赵启明特批。

### 3. 降级路由触发
- **触发条件**：主模型健康度低于阈值 / 连续超时 / 实例全不可用。
- **动作**：路由器按预设链路把流量切到备用模型（如大模型 → 蒸馏小模型）。
- **注意**：降级会改变输出质量，**必须在 Oncall 群同步受影响产品线接口人**（灵犀找陈昊）。降级恢复需人工确认主模型稳定后摘除降级标记，**部分模型的降级链路尚未配齐，缺省会直接返回 503，需逐个补全**。

## 三、历史踩坑记录

### 坑①：GPU 显存 OOM —— 长上下文请求打爆 KV-Cache
- 灵犀某次放开 128k 上下文后，少量超长请求把 KV-Cache 显存吃满，导致同实例其他请求被连带 OOM kill。
- **教训**：vLLM 的 `max_model_len` 与 `gpu_memory_utilization` 要按最坏情况留余量；超长上下文模型应单独成组，别和短请求混部。
- 显存调参细节找乔宁。

### 坑②：批处理（continuous batching）下的长尾放大
- 一个超长 generation 请求会占住 batch slot，拉高同批短请求的 TTFT。
- **缓解**：对输出长度设上限、长短请求分流到不同服务组。具体分流阈值仍在调，未固化。

### 坑③：冷启动导致首批请求超时
- 新模型 / 扩容新 pod 加载权重期间（大模型可达数分钟），未就绪就被路由进流量，首批请求全超时。
- **教训**：必须等就绪探针（readiness probe）通过再纳入路由；灰度放量要等权重 warm 完。

### 坑④：计费用量与网关用量对不齐
- 流式请求中断时，completion token 统计偶发漏记，导致计费用量低于网关侧统计。
- **现状**：T+1 对账脚本兜底修正，根因（流式中断的 token 计数）尚未彻底定位，**待排查**。

### 坑⑤：降级链路缺失直接 503
- 见上文「降级路由」——部分模型没配备用模型，主挂即 503。补全工作进行中。
