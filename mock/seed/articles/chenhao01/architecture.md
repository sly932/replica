# 灵犀系统架构总览

> 本文是灵犀对话式智能客服后端的架构主线文档，新人和值班同学先读这篇建立全局认知，再按需深入各模块的子文档。

## 1. 系统定位

灵犀是星澜对外旗舰产品，面向商家提供对话式智能客服。一句话概括链路：**用户消息进来 → 对话引擎理解并生成回复 → 答得了就直接回，答不了就转人工坐席**。后端要保证这条链路在高并发下稳定、可降级、可观测。

## 2. 四大模块

整个后端拆成四个核心模块，彼此通过内部 gRPC + 消息队列解耦：

1. **对话引擎（dialog-engine）**：链路核心。负责会话状态管理、意图识别、知识召回、Prompt 编排与调用大模型生成回复。它是 QPS 压力最大的模块，单独部署、单独扩容。
2. **知识库服务（kb-service）**：商家上传的 FAQ、文档、商品资料经过切片与向量化后存这里。对话引擎通过它做 RAG 召回。底层向量库用的是自建的 Milvus 集群。
3. **转人工与坐席网关（agent-gateway）**：当对话引擎判定「答不出」或用户主动要求时，由它负责把会话转交给真人坐席，做坐席分配、上下文透传、会话接管。详见《灵犀转人工链路设计》（吴桐维护）。
4. **商家控制台后端（console-api）**：给商家配置机器人、看会话数据、管知识库的后台 API。

## 3. 关键数据流

```
用户消息
  → 接入网关(im-gateway，对接推送中台的群机器人通道)
  → dialog-engine：取会话上下文 → 意图识别 → kb-service 召回 → 调 MaaS 推理 → 生成回复
  → 命中兜底规则/低置信 → agent-gateway 转人工
  → 会话结束 → 埋点上报数仓
```

## 4. 跨组依赖（重要）

灵犀不是孤岛，强依赖四个中台/职能组，出问题先判断是不是依赖方的锅：

- **账号与权限中台组**：商家登录、坐席登录的 SSO 鉴权。接入问题找林晚（注意她下周起休产假，接口人转章远）。
- **消息/通知推送中台组**：群机器人消息的下发通道。回调失败、限流找周珉。
- **模型推理平台（MaaS）组**：对话大模型的推理与配额。模型超时、配额超限找赵启明，配额申请走《灵犀对接 MaaS 配额说明》。
- **数据仓库与数据治理组**：会话指标埋点。

## 5. 部署与稳定性概览

线上按多可用区部署，dialog-engine 与 agent-gateway 是独立伸缩单元。具体环境拓扑、灰度发布与回滚步骤见《灵犀部署手册》；线上告警分级、扩容限流操作见《灵犀线上值班 SOP》。

## 6. 设计上的几个取舍

- **对话引擎无状态化**：会话上下文存 Redis，引擎实例本身无状态，方便水平扩容。代价是 Redis 成为关键依赖，Redis 抖动会直接影响全链路。
- **转人工兜底优先于「硬答」**：宁可转人工也不让模型在低置信时乱答，这是产品（林悦）和后端共同定的红线。
- **召回与生成分离**：kb-service 只管召回不管生成，方便独立优化召回质量。
