# 转人工与知识沉淀产品逻辑

> 本文定义灵犀的核心飞轮：**答不出 → 转人工 → 坐席解答 → 知识回流沉淀**。这是整个产品的价值闭环，也是灵犀区别于普通「问答机器人」的根本。技术实现见后端陈昊、转人工链路见吴桐文档，本文只讲产品逻辑与判定规则。

## 一、为什么需要这条链路

机器人不可能一上来就什么都会。灵犀的设计前提是：**承认机器人会答不出，并把每一次答不出变成一次知识增长的机会**。答不出不是失败，是知识库的缺口被暴露出来；坐席补上的每一个答案，都让机器人下次更强。这就是飞轮。

## 二、「答不出」的判定标准

对话引擎在生成回复前，会先做知识检索与置信度评估。满足以下**任一**条件即判定为「答不出」(answer_not_found)：

1. **检索置信度过低**：topK 检索的最高相似度 `score < 0.62`（v3.4 阈值，可在机器人配置里按商家调）。
2. **模型自评不确定**：生成阶段模型返回 `confidence` 标记为 `low`，或命中「我不确定 / 建议咨询人工」类兜底模式。
3. **命中敏感/高风险意图**：如退款金额、投诉、法律纠纷等，强制不由机器人作答（意图清单维护在机器人配置）。
4. **用户连续 2 轮表达不满**：如「答非所问」「没用」「转人工」，触发主动让位。

判定结果写入会话事件 `event_type = answer_not_found`，并带上 `reason`（low_score / low_confidence / sensitive_intent / user_dissatisfied）。

## 三、转人工触发

判定答不出后，按机器人配置的**转人工规则**决策：

- **坐席在线**：进入待接入队列，按分流策略（轮询 / 技能组 / 最闲优先）分配。排队超时（默认 90s）触发推送提醒（周珉，推送中台群机器人）。
- **坐席离线/非工作时间**：展示兜底话术 + 留言收集，转为工单。
- **可机器人继续兜底的场景**：先给「我帮您转接，同时为您查询…」的过渡话术，避免冷场。

每一次转人工都生成一条 `handoff_record`，关联原始会话、answer_not_found 的 reason、用户原始问题（这条很关键，是后续沉淀的「问」）。

## 四、知识回流沉淀（飞轮闭环）

坐席在坐席系统答完后，这通会话就成了一条潜在知识。沉淀流程：

1. **抽取候选问答对**：系统把「用户原始问题」作为 Q，把「坐席最终有效回复」作为 A，生成一条候选 FAQ（状态 `pending`）。
2. **坐席确认/编辑**：坐席可一键「沉淀为知识」，并对 Q/A 做精简（口语转规范表述）。切片粒度建议见《灵犀商家 FAQ 知识库规范》(slug: faq-kb-spec)。
3. **入库与向量化**：确认后进入商家知识库，由 MaaS（赵启明）做 embedding，下次同类问题即可命中。
4. **效果回看**：会话分析里可看「由回流知识命中并解决」的占比，衡量飞轮转速。

## 五、待明确：坐席答案的审核与脏知识防控

这是当前**尚未拍板**的部分，也是最容易出问题的环节：

- 坐席答得对不对、是否带有个人临场承诺（如随口给的优惠），**直接回流会污染知识库**。
- 目前 v3.4 只做了「坐席手动确认才入库」这一道关，**是否需要二次审核（运营/质检复核）、由谁审、审核 SLA 多久、错误知识如何回滚下线——这些都暂未明确**。
- 候选问答对自动入库 vs 人工双审的边界、置信度阈值是否复用 0.62、敏感意图答案是否一律禁止沉淀，**这些边界场景待定，需找林悦确认**，预计在 v3.5「知识回流自动化 v1」前定稿。

在规则定稿前，运营江岚按「宁缺毋滥」原则人工把关；脏知识一旦发现，临时走知识失效流程下线（见 faq-kb-spec）。
