# 星澜科技 — Replica 数字分身平台 Mock 数据方案

> 用途：为「Replica 员工数字分身平台」黑客松 demo 提供一套丰富、自洽、有真实感的虚拟公司底座。评委会亲自下场体验，因此组织结构、角色、文档清单要能支撑「问分身 → 答不出转真人 → 真人答完沉淀回分身 → 分身可交接」的完整闭环。
> 本文件**只做方案/清单蓝图**，不写文档正文。正文按最后一节「优先级建议」分批生产。
> 最近更新：2026-06-27

---

## 一、公司概览

- **公司名**：星澜科技（Stellaris Tech）
- **定位**：企业级 AI 科技公司，三条对外智能产品线 + 一个对内基建与中台事业部 + 职能支持。
- **规模**：74 人，15 个组。
- **三条产品线**：灵犀（对话式智能客服，★demo 主舞台）· 磐石（数据平台 / BI）· 悟空（AI 编程助手）。
- **协作文化设定**：远程 + 飞书办公，知识散落在个人文档、群聊、值班表里——这正是 Replica 要解决的「知识在人脑、问到人就被打断、人走茶凉」的痛点土壤。
- **数字分身渗透设定**：每位员工都有一个分身；Tier1 角色的分身「喂养充分、人格鲜明」，Tier2「半成型」，Tier3「刚建档、知识稀薄」——用层级差异演示「越用越懂你」。

## 五、后续造正文的优先级建议

> 原则：先把 demo 主路径跑通，再补支撑链路，最后撑场面。分三批生产文档正文。

**第一批（必做，demo 主路径）**
1. 陈昊全部 6 篇（架构 / 部署 / 值班 SOP / 踩坑 / 转人工链路 / MaaS 配额）——主舞台，质量最高、最详尽。
2. 其余 5 名 Tier1 各自的核心 2–3 篇（对应锚点②③④⑤⑥ 会被问到的那篇优先）。
3. 灵犀组级对外知识 4 篇（白皮书 / 接入 SOP / FAQ / API）。
4. 关键依赖文档：账号中台《SSO 接入指南》、推送《群机器人接入指南》、MaaS《模型与配额申请 SOP》——支撑跨组场景。

**第二批（支撑转人工 / 跨组 / 培训 / 交接链路）**
5. Tier2 共 12 人的中等文档（每人 2–4 篇），重点是吴桐（转人工真人专家）、章远（交接接任）、温知夏（休假/离职流程）、石川（故障协调）。
6. L&D《新人 FAQ》《Mentor 制度》、HR《离职交接 SOP》《请假制度》——支撑 ④⑤。

**第三批（撑场面，可批量生成 / 占位）**
7. Tier3 约 56 人的单篇文档，**仅标题 + 一句话简介即可**，可程序化批量生成占位，无需精修。
8. 其余各组组级对外知识，按需补全。

**数据自洽校验清单（生成正文时遵守）**
- 锚点①③⑤⑥ 的「被问问题」必须能在对应 Tier1 文档里找到答案（或故意答不全以触发转人工）。
- 锚点② 的指标口径文档与实际口径**必须制造冲突**，且《变更记录》缺最近 3 次——这是「文档过时」的钩子。
- 锚点④⑤ 必须体现「mentor 缺位 / 负责人休假」的断档感，文档里留交接缺口。
- 组间依赖（第一节依赖图）在文档交叉引用里要体现：业务组文档提到「找 XX 中台」，引导评委体验「跨组找人」。
