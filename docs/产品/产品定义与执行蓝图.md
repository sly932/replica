# Replica · 产品定义与执行蓝图

> toB Agent 赛道 · 员工数字分身平台 · 单人 · 开发窗口 6/27 11:05 → 6/28 14:30
> 本文是项目纲领，所有决策以此为准。竞品依据见 `竞品调研/`。

---

## 0. 一句话定位

**Replica——员工数字分身平台：3 分钟创建你的数字分身，替你回答重复问题、答不出转真人、越用越聪明、越用越懂你；经验可沉淀、可交接、可流动。**

## 1. 核心认知（立项哲学，贯穿全篇）

1. **Agent 解决重复性问题，把决策留给人**——agent 可以提出方案，但最终判断交还人类。
2. **好的 agent 应该"了解"每一个用户**——个性化 / 记忆 / 用户数据链路，是通用 agent 的护城河。

## 2. 痛点场景（demo / PPT 的素材，越具体越能让评委秒懂）

专家、文档、群机器人、新人、离职/休假、跨组协作——六个真实场景，本质都是**"知识锁在少数人脑子里、人一忙就断档"**：

1. **专家被打断**：你是专家，但大部分时间在被业务方、组里新人打断。
2. **文档过时无人维护**：你配了文档，有人反馈某些内容过时，你去沟通确认了哪里过时、标了 TODO 想有空改——然后没了下文。
3. **群机器人无人盯**：你给业务组配了群机器人能自动回答，但不知道答得怎么样；去翻群聊、标了答错的和没答上的、记了 TODO——转头忘了。
4. **新人 landing 无人带**：组里新员工入职，mentor 在忙自己的事，无暇顾及。
5. **离职/休假断档**：维护某块业务的人离职或休假，相关问题只能查可能过时的文档；没文档时，其他业务咨询直接停滞。
6. **跨组协作找不到人**：业务组遇到问题/需求，不知道找哪个组合作，只能在群里问；而知道答案的人，可能被刷屏没看到。

## 3. 产品定义（完整愿景 · 五大能力）

1. **3 分钟创建员工数字分身**——低门槛自助创建。
2. **双路自进化，越用越聪明**：
   - (1) agent 不会的问题 → **真人回答** → 沉淀（human-in-the-loop）。
   - (2) 长链路推理得到的结论 → **自动沉淀并进入审批回路**（审批门控质量）。
3. **多层记忆，越用越懂你**：
   - (1) 和自己的分身聊天 → 自动沉淀**语义化记忆**（事实、偏好、判断）。
   - (2) 总结会话 → 沉淀**情景记忆**（谁问过什么、怎么答的）。
4. **分身转移**——经验具象化，可流动、可交接（离职/休假/轮岗时分身随岗位移交）。
5. **原生 IM**——平台自带问答 IM；未来可作为数字员工接入飞书、钉钉等企业 IM。

## 4. Roadmap（讲未来空间，体现纵深，PPT 用）

1. **组织级别的分身**（从个人分身升到组织 brain）。
2. **维护每个「提问者 ↔ 回答者」之间的关系记忆**。
3. **更个性化**：多方面提取记忆 / 用户偏好，构建用户画像，分身回答时获得更多上下文，更好地帮用户解决问题。
4. **Proactive**：结合用户画像与记忆，主动做出更多行为，提升效率与用户黏性。

## 5. 差异化（避开"换皮"质疑）

| 维度 | Viven | Lucius | Question Base / Starmind | **Replica** |
|---|---|---|---|---|
| 对内/对外 | 对内个人分身 | 对外单一 AI | 对内问答捕获 | **对内 + 每人一个分身** |
| 人格化分身 | ✅ | ❌ 单一AI | ❌ 纯问答路由 | **✅ 有人格** |
| 上手成本 | 8 周部署 | 接 IM 即用 | 中 | **3 分钟自助** |
| 答不出闭环 | 没讲清 | 转人工+草稿 | ✅ 已做 | **双路自进化**（真人答 + 推理沉淀审批）|
| 多层记忆/懂你 | 弱 | ❌ | ❌ | **✅ 语义+情景，越用越懂你** |
| 分身转移交接 | ❌ | ❌ | ❌ | **✅ 经验可流动交接** |

**不被说成"换皮"的立足点（三个组合拳，竞品都缺）：**
1. **每人一个有人格的分身**（Lucius/QB 都是单一 AI / 纯路由）——**「个体 vs 组织」是数据模型层面的品类差异，最难被说成换皮**。
2. **多层记忆——越用越懂你**（个性化护城河，QB/Starmind 没有"了解每个用户"这条线）。
3. **分身可转移交接**（离职/休假断档场景的独家解法，无人做）。

> ⚠️ **叙事红线（来自 QB 调研）**：Question Base 已把"答不出→转人工→verify→沉淀复用"做成显式产品功能，这是**它的主场**。所以——
> - **不要把"答不出转人工再沉淀"当唯一卖点**单讲，否则落入换皮叙事；
> - 我们飞轮的落点是 **「沉淀回个人分身、越用越像本人」**，不是"沉淀进公司共享手册"；
> - 主轴始终是 **每人一个有人格分身 + 越用越懂你 + 可交接**。
> QB：Slack 原生、全公司共享问答机器人，学的是"公司知道什么"而非"每个人是谁、会怎么答"。详见 `竞品调研/QuestionBase.md`。

## 6. MVP 范围（1 天只切一条链路 · 决定完成度 25%）

**必做（demo happy path）：**
- 文档上传 → 切块向量化入库（**原文表 + 向量表分离**）
- 分身问答：RAG 检索 + 人格化系统提示 + **答不出时诚实声明**
- **双路自进化之主路**：答不出 → 生成「待回答」→ 真人补答 → 一键沉淀回知识库 → 再问命中
- **多层记忆之轻量版**：和分身聊天时，自动抽取并落一条语义记忆，回答时召回（让"越用越懂你"可见）
- **OpenAPI 批量灌库接口**（建组织/项目组/角色/上传文档触发向量化），供脚本灌 mock

**亮点做（有余力，决定创新 30%）：**
- 长链路推理结论 → 自动沉淀 → 审批入库（双路自进化之第二路，可做最小版）
- 分身转移：一键把某分身的知识/记忆复制移交给另一账号（demo 一个按钮即可，叙事价值大）

**只讲不做（PPT/架构留接口）：**
- 飞书/钉钉接入、组织级分身、关系记忆、用户画像、proactive、混合检索权重调优

## 7. Demo 主线（评委下场 1 分钟跑通 · 5 步飞轮闭环）

> 评委扮「刚加入灵犀组的新人」或「隔壁磐石组要对接的人」，不打扰真人，直接问分身。

1. **问命中**：问「陈昊」分身"灵犀的部署流程？" → 基于部署手册回答 → 秀 **问人即得 + RAG**
2. **诚实+转人工**：追问文档没覆盖的"灰度发布 A/B 配置冲突怎么处理？" → 分身诚实说"我不确定，已转给陈昊本人并记为待回答" → 秀 **不胡说 + 转对的人**
3. **真人补答 + 沉淀**：切陈昊视角，收到待回答，一句话答 → 一键沉淀入库
4. **闭环命中**：再问同样问题 → 这次答出来了 → **最有冲击力的一幕**
5. **越用越懂你**：聊天中提到"我是磐石组的，对接偏数据口径" → 下次提问分身带着这条记忆回答 → 秀 **多层记忆**
6.（可选收尾）点一下「分身转移」按钮：陈昊休假，把分身移交给同组接替人 → 秀 **经验可交接**

## 8. 技术架构（沿用最熟、1 天最稳）

- **前后端**：Next.js（App Router + Route Handlers 当 API）
- **数据/向量/Auth**：Supabase（Postgres + pgvector + Storage）
- **Agent**：分身 = 系统提示（人格 + 边界）+ 工具集（知识检索 / 记忆检索 / 答疑沉淀 / 待回答推送）
- **原生 IM**：平台内自建问答聊天界面，**MVP 不接飞书/钉钉**
- **部署**：Vercel（公网链接，满足"可访问 Demo"硬交付）

**数据表设计（关键修正：原文与向量分离）：**
```
replicas        分身：id, owner_user_id, name, role, org, team, persona_prompt
articles        文章原文：id, replica_id, title, content(原文), source, status, created_at
chunks          RAG 向量：id, article_id, replica_id, chunk_text, embedding(vector)
memories        记忆：id, replica_id, type(semantic/episodic), content, embedding, created_at
pending_qs      待回答：id, replica_id, question, asker, status, answer, answered_at
knowledge_items 沉淀条目：id, replica_id, content, source(human/reasoning), approval_status
```
> 原文存 `articles`、向量存 `chunks`，检索召回走 `chunks` 再回查 `articles` 拿原文/出处。

**虚拟公司：星澜科技（一家 AI 科技公司，评委自己就在这种公司，代入最快）**

```

    │       │     文档：系统架构、部署手册、值班 SOP、踩坑记录
    │       ├── 林悦  产品经理
    │       │     文档：PRD、需求池、竞品分析、版本规划
    │       └── 王哲  前端
    │             文档：组件库规范、联调约定、发布流程
```

## 10. 评分对位（怎么把分拿满）
