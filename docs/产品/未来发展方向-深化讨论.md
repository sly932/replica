# Replica · 未来发展方向 — 深化讨论

> 本文整理"个性化 / 记忆 / Proactive / 组织级推理 / 企业级评测"几条 roadmap 方向的深入讨论，作为蓝图 `产品定义与执行蓝图.md` 第 4 节 Roadmap 的展开。结论用于指导路演 PPT 第 7 页与后续产品规划。
> 讨论日期：2026-06-28。

---

## 0. 一句话主线

**个性化的本质不是"给每个人微调一个模型"，而是"收集更多数据 → 融合成画像 → 喂给记忆与 proactive"；而沉淀下来的数据，最终训练的是组织/公司级的领域推理能力——让经验资产留在公司，而非留在个人。**

---

## 1. 个性化的三层拆分（为什么不走 per-person LoRA）

把"个性化"拆成三层，各用对的工具，避免一刀切去给每人训 LoRA：

| 要个性化的东西 | 对的工具 | 理由 |
|---|---|---|
| 知识 / 偏好（"懂你"） | **记忆 + RAG** | 可更新、可召回、可引用出处；便宜且与产品护城河一致 |
| 说话风格 / 语气（"像你"） | persona prompt；真要高保真才上**轻量 per-person 风格 LoRA** | 风格用少量数据即可迁移，这是 LoRA 少数合理用途 |
| 领域推理（"懂这家公司"） | **组织 / 团队级 RFT 或 LoRA** | 数据量足、成本被全员摊薄，ROI 高 |

### per-person LoRA 的成本账（结论：不划算）
- **算力不是主要成本**：7~8B 模型上的 LoRA，单卡几十分钟即可，便宜。
- **真正的坑**：
  1. **数据稀疏**：个人样本太少，要么没效果、要么过拟合到那几条。
  2. **服务成本是大头**：N 名员工 = N 个 adapter，要存储、版本管理、随数据漂移反复重训；多-LoRA 在线服务（S-LoRA 类）显存/延迟/运维复杂度陡增。
  3. **与护城河冲突**：微调进权重的知识**不能引用出处、不能干净地说"我不确定"、且会过时**，违背"答不出转真人 + 可追溯沉淀"的核心卖点。
- **结论**：知识/偏好走记忆+RAG；风格按需上轻量适配；**把微调的力气投到组织/团队级**。

## 2. 组织 / 公司级领域推理（核心杠杆）

- **数据来源即飞轮产物**：双路自进化沉淀出的「问题 → 真人答案」对，本身就是一份**组织级 RFT / 微调的训练集**。不需要给每人训模型，而是用全公司沉淀语料，把**组织级分身**越练越懂这家公司的术语、格式、常见推理路径。
- **战略价值（用户强调）**：经验资产沉淀在**公司/组织层**，而非个人层——**即使中层领导班子更替，经验资产仍留在公司**。这是 toB 的关键卖点：组织记忆与领域推理能力不随人走。
- **闭环**：自进化语料 → 组织级 RFT → 组织分身更强 → 产生更多高质量沉淀 → 再训练。反哺 roadmap「组织级分身」。

## 3. 记忆融合（多源记忆 · Proactive 的燃料）

- 把 **对话记忆 + 会议 + 工作文档 + 日历** 融合成**用户 / 角色画像**。
- 定位：它是 proactive 的**前置燃料**——先有融合画像，才谈得上主动行为。
- 对应 PPT 第 7 页：「更个性化」→ 改名 **「记忆融合」**。

## 4. Proactive 场景清单

> 核心范式（呼应立项哲学）：**Agent 提方案，决策交还人。** Proactive 一律是"建议级"，不替人做最终决定。

### 围绕会议 / 决策
- **会前简报**：会议开始前，读日历邀请 + 相关文档 + 上次纪要 → 生成"这会聊啥、你上次立场、待解决问题、相关数据"。
- **会中实时调研**（标杆场景）：识别"要做 X / 竞品怎么做 Y" → 静默跑调研与竞品对比，散会时摘要已就位、会后即可决策。
- **决策预备包**：检测到决策点 → 自动备齐"选项 + 数据 + 先例 + 风险 + 隔壁团队当时怎么选"。

### 围绕知识飞轮（直接打六大痛点）
- **知识保鲜预警**：依赖的文档/答案可能过时（引用代码变更、负责人离职）→ 主动提醒 owner 更新（痛点 2）。
- **离职/休假前知识抢救**：日历显示将长假 / HR 信号离职 → 主动"采访"抢先沉淀将流失的隐性知识（痛点 5）。
- **重复提问聚合**：同一问题第 N 次被问 → 主动起草 FAQ/文档，人一键确认（喂自进化）。

### 围绕人与协作（注意红线，见下）
- **新人主动陪跑**：按同岗新人此阶段常卡的点，提前推送（痛点 4）。
- **主动撮合 / 重叠预警**（⚠️ 可选）：发现多组在做重叠的事 → 提示对齐。

### 两条红线（toB 落地必须讲）
1. **信任与授权**：proactive 做错会毁信任；读会议/文档/日历是敏感数据。必须 **建议级 + 权限分级授权 + 最终人拍板**。
2. **服从组织策略（撮合/重叠预警专属）**：**内部赛马可能就是公司的有意策略**——故意让团队并行竞争。所以"撮合 / 重叠预警"**不默认开启、不强推**，做成**可配置开关**：要去重提效的组织打开、要赛马竞争的组织关闭，同公司不同层级策略可不同。定位为"可选能力，服从组织策略"。

## 5. 企业级评测 Benchmark + 动态优化系统

> 这既是工程纵深，也是护城河：越用数据越多、系统越准。

- **为什么需要**：企业部署 agent 不能凭感觉，要**可度量**。
- **核心指标**：回答准确率 / 答不出转人工率 / 检索召回命中率 / 沉淀有效率 / 用户采纳率 / 满意度。
- **数据来源**：
  - **Agent trajectory**（执行轨迹）：query → 检索 → 工具调用 → 回答 → 是否转人工 → 是否被采纳，全链路记录。
  - **用户反馈**：赞/踩、采纳、纠错。
- **优化闭环**：trajectory + 反馈 → 构建**企业自有 benchmark / 回归评测集** → 定位弱项 → 优化（prompt / 检索权重 / 工具设计 / 记忆策略 / 组织级 RFT）→ 再评测 A/B。
- **护城河**：每家公司沉淀自己的领域评测集 + 轨迹数据，形成 data flywheel，越用越准、越难被替代。
