# Replica · 需求规格（#1 需求细化）

> 配套纲领 `产品定义与执行蓝图.md`。本文聚焦：谁用、用来干什么、有哪些页面、关键流程怎么走、MVP 验收标准。
> 优先级：**P0 = demo 主线必须**；P1 = 有余力做、能加分；P2 = 只讲不做（PPT/架构留接口）。

---

## 1. 角色（Actors）

| 角色 | 说明 | demo 里谁扮 |
|---|---|---|
| **分身主人** | 员工，创建并训练自己的分身、补答、审批、交接 | 陈昊（后端负责人）|
| **提问者** | 同事/新人/跨组的人，向分身提问 | **评委**（扮新人或隔壁组）|
| **管理员/数据接入** | 建组织/项目组/角色、批量灌数据 | 我们（通过 OpenAPI 脚本）|

## 2. 功能清单（按角色 × 优先级）

### 2.1 分身主人
| 功能 | 优先级 | 说明 |
|---|---|---|
| 3 分钟创建分身 | **P0** | 填基本信息 + 岗位/角色 + 人格设定，生成分身 |
| 上传文档建知识库 | **P0** | 上传 → 入 `articles`（原文）→ 切块向量化入 `chunks` |
| 查看 & 补答「待回答」 | **P0** | 收到分身答不出的问题，一句话补答 → 一键沉淀回知识库 |
| 和自己分身聊天沉淀记忆 | **P1** | 对话中自动抽取语义记忆，落 `memories` |
| 审批自动沉淀的知识条目 | **P1** | 长链路推理结论 → 待审批 → 通过后入库 |
| 分身转移/交接 | **P2** | 一键把分身知识+记忆移交给接替账号 |

### 2.2 提问者
| 功能 | 优先级 | 说明 |
|---|---|---|
| 浏览/搜索分身 | **P0** | 按组织/项目组/角色找到要问的人的分身 |
| 向分身提问 | **P0** | RAG 检索 + 人格化回答 + **标注出处** |
| 看到"诚实说不会 + 转人工" | **P0** | 检索不到/低置信 → 分身声明不确定 + 生成待回答 |
| 收到补答后的回答 | **P1** | 本人补答后，提问者可见更新/再问命中 |
| 越用越懂你 | **P1** | 提问者身份/偏好被记忆，后续回答带上下文 |

### 2.3 管理员 / 数据接入（OpenAPI）
| 功能 | 优先级 | 说明 |
|---|---|---|
| 建组织/项目组/角色 | **P0** | REST 接口，供脚本批量建 mock 组织 |
| 批量上传文档触发向量化 | **P0** | REST 接口，灌 mock 文档 → articles + chunks |
| 组织知识看板 | **P2** | 谁该写 SOP、哪些知识单点风险（只讲）|

## 3. 页面 / 路由清单

| 路由 | 页面 | 优先级 | 职责 |
|---|---|---|---|
| `/` | 首页 | P0 | 产品一句话 + 进入分身广场 / 登录 |
| `/replicas` | 分身广场 | **P0** | 按组织/项目组浏览 + 搜索分身（评委从这进）|
| `/replica/[id]` | **分身问答页** | **P0** | 核心：聊天式问答 + 出处 + 转人工提示（demo 主场）|

| `/me/pending` | 我的待回答 | **P0** | 补答 + 一键沉淀 |

| `/me/memories` | 我的记忆 | P1 | 查看分身对我/提问者的记忆 |

## 4. 关键流程时序（核心 3 条 + demo 闭环）

**流程 A · 提问 → RAG → 答 / 转人工**（P0）
```
提问者发问 → 向量化 query → 检索 chunks(top-k) → 回查 articles 取原文/出处
  → 命中且置信足够 → 人格化 LLM 生成回答 + 出处
  → 检索不到 / 置信低 → 分身诚实声明"不确定" → 生成 pending_q 推送本人
```

**流程 B · 待回答 → 补答 → 沉淀 → 再问命中**（P0，飞轮闭环，demo 高光）
```
本人在 /me/pending 看到问题 → 一句话补答 → 系统将「Q+本人答」写入 articles + 向量化
  → 提问者再问同样问题 → 这次命中、答出来
```

**流程 C · 创建分身 → 上传文档 → 向量化**（P0）
```
填分身信息(角色+人格) → 上传文档 → 存 articles → 切块 embedding → 存 chunks → 可被检索
```

**流程 D · 聊天沉淀记忆 → 回答带上下文**（P1，"越用越懂你"）
```
对话中识别可记忆信息(身份/偏好/事实) → 抽取为 memory → 落 memories
  → 后续提问时召回相关 memory 注入上下文 → 回答更贴合此人
```

## 5. 分身回答的"边界"设计（技术深度 & 不胡说）

- 检索结果为空或最高相似度 < 阈值 → **不强行编**，走转人工。
- 回答必须**带出处**（来自哪篇 article / 哪条沉淀），可点开看原文。
- 人格化只影响**语气/口吻**，不影响事实来源（事实只来自检索到的知识/记忆）。

## 6. MVP 验收标准（goal-driven，对应 demo 主线 5 步）

- [ ] 评委能从 `/replicas` 找到陈昊分身并提问，得到**带出处**的回答（流程 A 命中）
- [ ] 追问文档外的问题，分身**诚实转人工**并生成待回答（流程 A 兜底）
- [ ] 陈昊在 `/me/pending` 补答 → 一键沉淀（流程 B 写入）
- [ ] 再问同一问题 → **这次答出来**（流程 B 闭环，高光时刻）
- [ ] 对话中提到的身份/偏好，下次提问被**带上下文**回答（流程 D，可降级为 P1）
- [ ] OpenAPI 能把 mock 组织 + 文档批量灌进库（数据接入 P0）

## 7. 暂不做（P2，只在 PPT/架构讲）
飞书/钉钉接入、组织级分身、关系记忆、用户画像、proactive、知识看板、混合检索权重调优、分身转移（如时间紧）。
