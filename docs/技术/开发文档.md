## 1. API 设计

所有 API 在 `web/app/api/` 下，薄层：鉴权 → 校验 → 调 `lib/` → 返回。

### 1.1 流式问答（核心）
```
POST /api/chat/stream            （SSE）
body: { replicaId, askerId, text, images?: [{data,mimeType}], threadId? }
→ SSE 事件流：
   event: delta   data: {"text":"..."}      逐字
   event: source  data: {"title","articleId"} 命中出处
   event: handoff data: {"pendingId","msg"}   答不出转人工
   event: done    data: {"messageId"}
```

### 1.2 分身 / 资料
```
GET  /api/replicas               列表（按组织/项目组/角色）
GET  /api/replicas/:id           详情（含 profile）
POST /api/replicas               创建（3 分钟建分身）
PATCH /api/replicas/:id          改可编辑字段（gender/bio/hobbies/mbti）
```

### 1.3 知识来源（articles）
```
GET  /api/replicas/:id/articles          文档列表
POST /api/replicas/:id/articles          上传：{ title, sourceUrl } → 入 articles + 切块向量化入 chunks
PATCH /api/articles/:id                   启用/停用（enabled）
```

### 1.4 记忆 / 知识条目
```
GET  /api/replicas/:id/memories          通用+领域记忆
PATCH /api/memories/:id                   编辑 / 启停
GET  /api/replicas/:id/knowledge         沉淀条目
PATCH /api/knowledge/:id                  审批（approved）/ 启停
```

### 1.5 待回答（飞轮主路）
```
GET  /api/replicas/:id/pending           待回答列表
POST /api/pending/:id/answer             本人补答 → 写回 articles + 向量化 → 标 answered
```

### 1.6 图片上传（多模态）
```
POST /api/upload                         multipart → Supabase Storage → { url }
```

### 1.7 数据接入（OpenAPI 批量灌库）
```
POST /api/ingest/org                      建组织/项目组/角色（分身）
POST /api/ingest/articles                 批量上传文档（触发向量化）
→ 供 mock 灌库脚本调用，按 mock/mock-plan.md
```

## 2. pi-agent-core 集成方案（lib/agent/）

### 2.1 ReplicaAgent 封装
```ts
// lib/agent/replicaAgent.ts（伪代码）
import { Agent } from '@earendil-works/pi-agent-core'
import { createProvider } from '@earendil-works/pi-ai'

// cloudsway = OpenAI **Responses API** 协议（非 chat/completions）；pi-ai 用 openai-responses provider 接
// baseUrl=https://api.agentsway.dev，Bearer sk-key；embedding 走独立服务（见 §5）
const provider = createProvider({ api: 'openai-responses', baseUrl: process.env.CHAT_BASE_URL, apiKey: process.env.CHAT_API_KEY })
// 模型由前端对话框传入（cloudsway 多模态：claude-4.x / gemini-3.x / gpt-5.x）

export function buildReplicaAgent(replica, ctx) {
  const agent = new Agent({
    provider, model: process.env.MODEL,
    systemPrompt: buildPersonaPrompt(replica),   // 人格 + 边界
    tools: [searchKnowledge, searchMemory, handoffToHuman, saveInsight],
  })
  return agent
}
```
- **systemPrompt**：注入 role/bio/mbti + "只依据检索到的知识与记忆回答，答不出就调用 handoffToHuman，绝不杜撰"。
- **长期记忆**：召回的 memory + RAG 结果，经 `transformContext` 或拼进 systemPrompt 注入（pi 只自带短期 transcript）。

### 2.2 工具集（按场景动态加载 — context engineering）
> **动作空间按场景裁剪**：和**自己**分身聊天才挂「记忆管理」；回答**别人**才挂「沉淀问题/沉淀长链路」。

| tool | 加载场景 | 输入 | 行为 / 返回 | 落库 |
|---|---|---|---|---|
| `manageMemory` | 仅「主人 ↔ 自己分身」 | **type**(add\|update\|delete\|list) + 随 type 参数（add:content；update:id,content；delete:id） | 管**主人**记忆(owner 校验)。**add 自动向量查重**：命中相似则返回 `{success:false, conflicts:[{id,content}]}`（agent 据此请主人定夺，策略见 systemPrompt）；delete=软删(deleted)，**不软失效** | memories |
| `searchKnowledge` | 通用 | query | 向量+BM25 同时召回 **文档块 ∪ 知识条目** → 返回 [**type**(chunk\|knowledge), 简介(chunk.context／条目摘要), 内容, **id**(文档id／条目id)] | 读 chunks + knowledge_items |
| `readDocument` | 通用 | docId, curIdx, readLen | 分页读文档：单次返回 ≤ MAX 字符，按 curIdx+readLen 取段，返回该段 + 下一 curIdx | 读 articles |
| `readKnowledgeItem` | 通用 | itemId | 返回知识条目**全文**(question+answer) | 读 knowledge_items |
| `saveQuestion` | 回答别人时 | question | 答不出 → 沉淀问题到知识条目（待回答），真人后续回答 | knowledge_items(pending_answer) |
| `saveInsight` | 回答别人时 | question, conclusion | 长链路（查多篇文档综合推理）结论 → 沉淀知识条目（待审批） | knowledge_items(pending_review) |
| `searchConversation` | 通用 | query, before?/after?(ISO), limit(默认5), sort_order? | 会话 summary **向量+BM25** 召回，可选时间窗过滤/排序（合并"内容召回+时间召回"为一个工具，因主轴是语义召回、时间仅过滤） | 读 conversations |

> 工具失败一律 `throw`（pi 约定）。`manageMemory` 严格 owner 校验：非主人会话不加载该工具。
> 记忆的**存储/提取/召回/冲突处理**细节待 `竞品调研/记忆组件调研.md` 回收后定稿。

### 2.3 流式
`agent.subscribe(e => { if(e.type==='text_delta') write(sse,'delta',e.text) })`；参考 pi `streamProxy`。

### 2.4 多模态

## 4. 关键数据流

```
建分身(profile) → 上传文档 → 切块 embedding 入 chunks
提问(text+图) → ReplicaAgent.prompt
   → searchKnowledge/searchMemory 注入上下文
   → 命中: 流式回答 + 出处
   → 不命中: handoffToHuman → pending_qs(SSE handoff 事件)
本人补答(pending) → 写回 articles + 向量化 → 再问命中（飞轮闭环）
```

## 5. 依赖与注意事项

- **运行时**：Node ≥ 22（pi-agent-core 要求）。
- **包**：`@earendil-works/pi-agent-core`、`@earendil-works/pi-ai`、`@supabase/supabase-js`、`next`、markdown 渲染（react-markdown）。
- **环境变量**（chat 与 embedding 分离，cloudsway 只有 chat）：`SUPABASE_URL` / `SUPABASE_ANON_KEY` / `SUPABASE_SERVICE_KEY` / `CHAT_BASE_URL` / `CHAT_API_KEY` / `CHAT_MODEL`（cloudsway，多模态）/ `EMBEDDING_BASE_URL` / `EMBEDDING_API_KEY` / `EMBEDDING_MODEL` / `EMBEDDING_DIM`（独立 embedding 服务）。
- **流式**：Railway 常驻部署（非 Vercel serverless），避免 SSE 被超时切断。
- **向量**：embedding 维度需与 pgvector 列定义一致；建 ivfflat 索引。
- **多模态**：图片 base64 体积大，上传走 Storage 存 URL，prompt 时再取；注意请求体上限。
- **pi 约定**：工具失败 throw；streamFn 失败编码进流、别 reject。
- **lib 解耦**：lib/ 不 import 任何 next/* ，保证将来可拆独立服务。
