# Replica · 技术架构

> 员工数字分身平台 · 黑客松 MVP。产品定义见 `../产品/`，pi-agent-core 调研见 `./pi-agent-core调研.md`。
> 技术栈：**Next.js 全栈 + lib 层解耦 + Railway 部署 + Supabase 数据 + pi-agent-core 分身内核**。

---

## 1. 技术栈总览

| 层 | 选型 | 作用 |
|---|---|---|
| 客户端 | **Next.js（App Router）** | web 前端 + API 一体；未来 Tauri 复用 |
| 后端逻辑 | **`lib/` 独立模块** | pi-agent-core 封装 / RAG / 记忆 / 飞轮；与框架解耦，将来可拆独立服务 |
| Agent 内核 | **pi-agent-core** (`@earendil-works/pi-agent-core`) | 分身大脑：对话状态、工具、流式、多模态 |
| LLM 接入 | **pi-ai + OpenAI 兼容中转（cloudsway）** | `createProvider({baseUrl})`，多模态模型 |
| 数据 | **Supabase**（Postgres + pgvector + Auth + Storage） | 结构化数据 / 向量检索 / 图片存储 |
| 部署 | **Railway**（常驻 Node） | 跑 Next standalone；长 SSE 流式 + 定时任务不受 serverless 限制 |

## 3. 选型理由

- **Next.js 全栈**：用户熟悉其目录结构；前后端一体、开发快；App Router 的 route handler 原生支持流式 `Response`。
- **lib 层解耦**：pi-agent-core 调研结论"后端逻辑重时独立服务更省心"。把重逻辑放 `lib/`、route handler 只做薄层 → 既享 Next 一体化，又保留"将来拆独立服务 / 供 Tauri 共用"的退路，零返工。
- **Railway 而非 Vercel**：常驻 Node，长 SSE 流式与 `node-cron` 定时任务不受 serverless 超时限制。
- **Supabase**：一站式 Postgres + pgvector + Auth + Storage，单人黑客松最省基建。
- **pi-agent-core**：框架无关 TS 库，流式 first-class（`streamProxy`），原生多模态图片，可接 cloudsway 中转。

## 5. 数据模型（Supabase · 原文与向量分离）

```sql
replicas        分身：id, owner_user_id, name, mis_id, role, org, team,
                persona_prompt, gender, bio, hobbies, mbti, created_at
articles        文章原文：id, replica_id, title, content, source_url, status, created_at
chunks          RAG 向量：id, article_id, replica_id, chunk_text, context,
                embedding vector, idx     -- context=上下文化前缀(Contextual Retrieval)；embed(context+chunk_text)
memories        记忆：id, replica_id, kind('semantic'|'episodic'), content,
                source_qa_id, embedding vector, enabled, deleted, created_at
                -- agent 主动写入；source_qa_id 溯源到产生它的问答；delete=软删(deleted)，不软失效
conversations   会话：id, replica_id, asker_id, direction('ask_me'|'i_ask'),
                summary, summary_embedding vector, created_at     -- summary 向量化(工具6)
messages        消息：id, conversation_id, role, content, images(jsonb), created_at
knowledge_items 知识条目（统一表，含待回答，**向量化**）：id, replica_id, question, answer,
                embedding vector,        -- 条目也向量化，与 chunks 一起进 searchKnowledge 检索池
                source('human'|'reasoning'),
                status('pending_answer'|'pending_review'|'approved'|'archived'),
                deleted, created_at
-- Storage bucket: chat-images（多模态图片，存 URL 引用）
```
> - 知识检索：query 向量化 → 查 `chunks`(pgvector) + BM25 → 回查 `articles` 取原文/出处。
> - 对话检索：query → 查 `conversations.summary_embedding` + BM25（可加时间范围）。
> - `pending_qs` 已并入 `knowledge_items`（status=pending_answer 即"待回答"）。删除一律软删（deleted）。

## 6. 关键技术点

### 6.1 pi-agent-core 集成（lib/agent/）
- 每个分身一个 `Agent` 实例（`ReplicaAgent` 封装）。
- **systemPrompt** = 人格（profile：role/bio/mbti）+ 边界（"答不出诚实转人工，不杜撰"）。
- **工具集 tools**：`searchKnowledge`（RAG）、`searchMemory`、`handoffToHuman`（生成 pending_q）、`saveInsight`（推理沉淀→审批）。
- **provider**：`createProvider({ baseUrl: cloudsway, apiKey })`，多模态模型。
- 长期记忆 pi 不自带 → 我们在 `transformContext` / systemPrompt 注入召回的 memory + 检索结果。

### 6.2 RAG（lib/rag/）— Contextual Retrieval
分块后，对每块用 LLM 基于**整篇文档**生成 50–100 token 的「上下文说明」存 `chunks.context`，prepend 到原块；对 **context+chunk_text 一起** 做 embedding（contextual embedding）+ BM25（contextual BM25），RRF 融合 → 回查 `articles` 出处；低于阈值触发转人工。
> ❌ 不用 summary：Anthropic 实测"文档摘要/摘要索引"收益有限；contextual（补全块的背景指代）才有效（失败率 -49%，+rerank -67%）。
> ✅ **知识条目同样向量化**，与 chunks 一起进入 `searchKnowledge` 的检索池。

### 6.3 流式 SSE（app/api/chat/stream/）
参考 pi 的 `streamProxy`：后端持 key、`agent.subscribe('text_delta')` → 写入 SSE `Response`；前端 `EventSource` 逐字渲染。

### 6.4 多模态（图片）
前端选图→预览→base64；`agent.prompt(text, images: ImageContent[])`；图片 URL 存 Storage、消息存 `messages.images`。

### 6.5 多层记忆（lib/memory/）
**两通道**：
- **语义记忆**(kind=semantic) = 原子事实（一句话），**agent 主动调 `manageMemory` 写入**（仅主人↔自己分身场景），每条 `source_qa_id` 溯源。
- **情景记忆**(conversations) = 会话 summary，append-only，向量召回，**不做冲突合并**（历史快照）。
**召回**：常驻画像（资料 role/bio/mbti + 稳定事实）直接注入 systemPrompt；长尾事实/情景按需 `searchMemory`/`searchConversation` 向量召回，带时间元数据。
**冲突处理（仅语义，不软失效）**：`add` 时自动向量查重 → 命中相似则 add 失败、返回冲突 id → agent 在聊天中**告知主人 + 建议 update 或 delete+add，由主人决定**（不自动覆盖、不软失效）。
**飞轮**：沉淀环（每轮 Q&A → 情景留档 + agent 记语义）做；进化环（空闲 sleep-agent 聚合重写）MVP 不做、只讲。
> 设计依据见 `../竞品调研/记忆组件调研.md`（原子事实=mem0/A-MEM 共识；冲突决策借 mem0 但落库不软失效，按本项目要求物理 update/delete）。

### 6.6 自进化飞轮（lib/flywheel/）
答不出→`pending_qs`→本人补答→写回 `articles`+向量化→再问命中；复杂推理→`knowledge_items`(pending)→审批→enabled。

### 6.7 定时任务（Railway）
`node-cron`：待回答推送、记忆整理、（未来 proactive）。

## 7. 部署拓扑
- **Railway**：Next.js standalone（`next start`）常驻；环境变量注入 Supabase / cloudsway key；`node-cron` 同进程或独立 worker。
- **Supabase**：托管 Postgres + pgvector + Storage。
