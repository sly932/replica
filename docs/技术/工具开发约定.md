# Replica Agent 工具开发约定

> 共享基础层 + 样板工具（searchKnowledge）已落地并真跑通。后续 7 个工具一律照此写。
> 相关：`docs/技术/pi集成速查.md`（§3 工具）、`docs/技术/开发文档.md`（§2.2 工具表）、`docs/技术/技术架构.md`（§6.2 RAG）。

## 1. 分层与已有地基

```
lib/db/client.ts      → supabaseAdmin（service_role，仅 server）
lib/db/queries.ts     → 向量检索(RPC) + 关键词检索(pg_trgm) + RRF + CRUD helpers   ← 新增
lib/rag/index.ts      → chunkText / genContext / ingestArticle / hybridSearch      ← 新增
lib/llm/embedding.ts  → embed(string[]) / embedOne(string)（ofox，1536 维）
lib/llm/chat.ts       → buildChatModel / streamFn（cloudsway，max_steps:1）
lib/agent/tools/_shared.ts        → ToolCtx / okResult / fail / Type / Static      ← 新增
lib/agent/tools/<name>.ts         → 每个工具一个 make<Name>Tool(ctx) 工厂           ← searchKnowledge 为样板
lib/agent/tools/__tests__/*.test.ts → 集成测试（真连 Supabase/cloudsway/ofox）
```

**铁律**：`lib/` 不 import 任何 `next/*`（将来可拆独立服务）。工具只 import `./_shared` 拿 typebox 与返回/错误 helper，不要各处直连 pi-ai。

## 2. 工具工厂签名（统一形态）

工具是**工厂**，不是单例：`make<Name>Tool(ctx: ToolCtx) => AgentTool<Schema>`。
工厂把运行期上下文（`replicaId` / `isOwner`）闭包进 `execute`。按场景裁剪动作空间时，由 `lib/agent` 装配层决定挂哪些工具（如 `manageMemory` 仅 `isOwner` 时挂）。

```ts
import type { AgentTool } from '@earendil-works/pi-agent-core'
import { Type, type Static, okResult, fail, type ToolCtx } from './_shared'
import { hybridSearch } from '../../rag'

const Params = Type.Object({
  query: Type.String({ description: '检索关键词或问题' }),
  topK: Type.Optional(Type.Number({ description: '返回条数，默认 6', default: 6 })),
})

export function makeSearchKnowledgeTool(ctx: ToolCtx): AgentTool<typeof Params> {
  return {
    name: 'search_knowledge',           // 模型看到的工具名（snake_case）
    label: 'Search Knowledge',          // UI 显示
    description: '在该分身的知识库中检索相关内容…',
    parameters: Params,                 // typebox schema
    execute: async (_toolCallId, params: Static<typeof Params>) => {
      const query = params.query?.trim()
      if (!query) fail('BAD_INPUT', 'query 不能为空')
      const results = await hybridSearch(ctx.replicaId, query, params.topK ?? 6)
      return okResult(text, { count: results.length, results })
    },
  }
}
```

## 3. 统一返回 / 错误约定（来自 `_shared.ts`）

- **成功**：`return okResult(text, data)`
  → `AgentToolResult`：`{ content:[{type:'text',text}], details:{ ok:true, code:'OK', data } }`
  - `text`：给**模型**看的正文。
  - `details.data`：给**前端/日志**的结构化结果（`details` 固定 `{ ok, code, data }` 三件套，统一解析）。
- **失败**：`fail(code, msg)` → `throw new Error('[CODE] msg')`（pi 约定：失败必须 throw，**绝不**把错误塞进 content）。
  - `code` 用稳定大写下划线错误码（`BAD_INPUT` / `NOT_OWNER` / `NOT_FOUND` …），`msg` 人读。
- schema 用 **typebox**（`Type` / `Static` 从 `_shared` 再导出，底层来自 `@earendil-works/pi-ai`）。**不是 zod、不是裸 JSON Schema**。

## 4. 怎么调 lib/db + lib/rag

- **检索类工具**（searchKnowledge / searchConversation / searchMemory）：优先用 `lib/rag` 的 `hybridSearch`，或 `lib/db/queries` 的 `match*`（向量）+ `keyword*`（pg_trgm）+ `rrfFuse`（k=60）自行组合。
  - 向量检索统一走 `functions.sql` 里的 RPC（`supabaseAdmin.rpc('match_xxx', {...})`），supabase-js 不能直接做 pgvector 查询。
  - 关键词检索用 `pg_trgm` 的 `ilike`（`queries.ts` 的 `keyword*`，按空白切词 OR）。
- **写入类工具**（manageMemory / saveQuestion / saveInsight）：用 `insertRow` / `updateRow` / `softDelete`（软删 = `set deleted=true`，**不软失效**）。
- **embedding**：`embedOne(query)` / `embed(texts)`（ofox，1536 维，与 schema `vector(1536)` 一致）。
- **RAG 入库**：`ingestArticle(replicaId, title, content, sourceUrl)`，内部 `chunkText`（~500 字/块）→ `genContext`（cloudsway 非流式 `/v1/responses`，`SUMMARY_MODEL`，`max_steps:1`，50-100 字上下文说明）→ `embed(context + chunk_text)`（Contextual Retrieval）→ 入 chunks。

## 5. 测试怎么写、怎么跑

每个工具配一个 `lib/agent/tools/__tests__/<name>.test.ts` 集成测试（**真连** Supabase/cloudsway/ofox，不 mock）：

1. `crypto.randomUUID()` 造独立 `replicaId`，`insertRow('replicas', { id, name, mis_id: 'test-…' })`（`mis_id` 带 uuid 后缀避免唯一冲突）。
2. 准备数据（如 `ingestArticle` 灌 `mock/seed/articles/<mis_id>/*.md`）。
3. `make<Name>Tool(ctx).execute(...)`，断言 `details.ok` / `details.data` 与正文关键词。
4. `finally` 里删 test replica（`replicas` 外键 cascade 自动清 articles/chunks）。

**运行命令**（需加载 `web/.env.local`）：

```bash
cd web
npx tsx --env-file=.env.local lib/agent/tools/__tests__/searchKnowledge.test.ts
# 或用脚本：npm run test:tool -- lib/agent/tools/__tests__/searchKnowledge.test.ts
```

两个本地环境注意点（**仅本地测试**，生产 Railway 直连不受影响）：

- **ESM 解析**：`web/lib/package.json` 标了 `{ "type": "module" }`，让 tsx 把 `lib/` 下 `.ts` 当 ESM 编译——否则 tsx 默认编成 CJS，`require()` 仅含 `import` 条件的 pi-ai/pi-agent-core 会报 `ERR_PACKAGE_PATH_NOT_EXPORTED` / require(esm) cycle。（`lib/` 全是 ESM 语法的 `.ts`，无 `.js`，不影响 Next 编译。）
- **代理**：本机若开了 clash 等代理（`HTTP(S)_PROXY`），Node 22 的 `fetch` 默认不读代理环境变量，而 ofox（embedding）本地需走代理。测试文件顶部据此用 `undici` 的 `setGlobalDispatcher(new EnvHttpProxyAgent())` 兜底（仅当检测到代理环境变量时）。
