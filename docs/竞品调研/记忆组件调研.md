# Agent 记忆组件调研

> 面向「员工数字分身平台 Replica」的记忆系统设计参考。
> 调研对象：mem0、MemPalace、Letta（前 MemGPT）+ Zep/Graphiti、A-MEM、MemoryOS。
> 更新日期：2026-06-27。每个竞品只讲技术实质，按 5 个关键维度对比：**存储 / 提取 / 召回 / 冲突处理 / 集成方式**。

---

## 0. 一句话技术实质

| 竞品 | 技术实质（剥掉宣传语） |
|---|---|
| **mem0** | 把对话用 LLM 抽成**一条条原子事实**，写入前再用 LLM 做 **ADD/UPDATE/DELETE/NOOP** 决策，避免重复与矛盾。冲突处理是它的核心卖点。 |
| **MemPalace** | "记忆宫殿"是营销隐喻，本质 = **逐字 chunk + 单个 ChromaDB collection 向量检索**，叠一张带有效期但不检测矛盾的 SQLite 三元组表，**全程无写入期 LLM 抽取**。 |
| **Letta (MemGPT)** | 把 LLM 当操作系统：**core memory 常驻上下文 + recall/archival 分页换出**，由 agent **自己调用 memory 工具**编辑记忆，靠"睡眠时 agent"异步整理。 |
| **Zep / Graphiti** | **双时间轴时序知识图谱**：LLM 抽实体与关系边，新事实让旧边"失效（invalidate）"而非删除，混合检索（向量+BM25+图遍历）。 |
| **A-MEM** | Zettelkasten 卡片盒：每条 note 由 LLM 即时生成 keywords/tags/context，**自动建链接**，新笔记加入后**回溯重写**旧笔记（memory evolution）。 |
| **MemoryOS** | "记忆操作系统"：**STM/MTM/LPM 三级分层**，靠 **FIFO + 热度公式**做分页换出与晋升（冷页换出、热页升入长期画像）。 |

## 1. mem0（重点：冲突处理机制）

**实质**：一个可插拔的"记忆层"，把每轮对话（user+assistant 消息对）经**两阶段流水线**沉淀为原子记忆。最大特征是写入前的 LLM 决策机制，而非无脑 append。

- **存储**：粒度是**原子事实（atomic fact / memory）**——一句话级别的自然语言陈述（如"用户喜欢喝美式咖啡"）。主存于**向量库**做语义检索；可选 `Mem0g` 变体额外建**有向标注图**（节点=实体含 type/embedding/metadata，边=三元组 source-relation-destination）做关系建模。不强分语义/情景类型，靠 metadata + 时间戳承载。
- **提取**：**两阶段**。① Extraction：LLM 结合「滚动会话摘要 + 最近若干条消息」抽出候选事实（salient facts）。② Update：见下。属**异步后处理**思路（不阻塞主回答）。
- **召回**：候选事实 / query 走**向量相似度**检索 top-k 历史记忆；Mem0g 额外做图检索。检索结果作为上下文拼进 prompt。
- **冲突处理（核心）**：Update 阶段对每条候选事实，先向量检索出最相似的历史记忆，再把「候选 + 相似历史」一起喂给 LLM，让它**通过 tool call 在 ADD / UPDATE / DELETE / NOOP 四选一**：全新→ADD，是补充/细化→UPDATE，与旧记忆矛盾→DELETE 旧的，无变化→NOOP。**决策完全交给 LLM**，mem0 不做额外的产品级编排规则。这是它区别于"只会 append"的记忆库的关键。
- **集成**：OSS 自托管库 + Mem0 Platform 托管云，**同一套 SDK/REST API**（Python/TS）。核心方法 `add / search / get_all / update / delete / batch*`。多租户作用域用 **`user_id` / `agent_id` / `run_id`** 过滤（v3 把它们放进 `filters`）。后端 LLM、embedding、向量库、图库均可替换。

## 2. MemPalace

**实质**：本地、极简的开源记忆系统（作者 recca0120），主打"逐字存储、零写入期 LLM 调用"。第三方独立审计指出其"宫殿/空间记忆"层级**主要是营销隐喻**，跑分高来自原始向量检索而非空间结构。技术栈：ChromaDB + SQLite + PyYAML。

- **存储**：粒度 = **drawer**（每条消息/文件块**逐字**存一条），全部塞进**单个 ChromaDB collection**，靠 metadata 区分。Wing→Room→Hall→Closet→Drawer 五级层级是**纯 metadata 驱动、查询时现算**，非强制结构。另有 SQLite 时序图（entities / triples 两表，triple 带 `valid_from/valid_to`）。按规则分 5 类（决策/偏好/里程碑/问题/情绪）。读取分 L0–L3 渐进加载（L0 身份 ~50-100 token → L3 全量语义检索）。
- **提取**：**全规则、零 LLM**。正则抽实体、词频打分、阈值分类；只做逐字存储 + 缩写压缩（AAAK 实体码映射），不做语义摘要。时机：异步/写入期，靠 Claude Code / Cursor 的 auto-save hook 周期触发。
- **召回**：实质=**纯向量最近邻**（ChromaDB）+ metadata 范围过滤；知识图谱只支持平铺 triple 查找，**无多跳遍历**。可选 LLM rerank（top-20 选 1）。`wake-up` 把 top-k 块拼进 system prompt 前。
- **冲突处理**：**很弱**。仅文件级去重 + 入库前向量相似度查重；图谱**不检测矛盾**，矛盾事实静默堆积，仅支持手动 invalidate（置 valid_to）表示过时。**无 add/update/merge 决策逻辑。**
- **集成**：主走 **MCP**（20-35 个 tool）；CLI（`mine/sweep/search/wake-up`）；可插拔后端的 Python API。`uv tool install mempalace`。

> ⚠️ 警示：官方 LongMemEval 96.6% 实为**未压缩原始向量检索**的成绩，与"宫殿结构"无关；开启压缩后掉到 84.2%。对它的层级宣传要打折看。

## 3. Letta（前 MemGPT）

**实质**：把 LLM 当"操作系统"——上下文窗口是"内存(RAM)"，外部存储是"磁盘"，agent 通过工具调用在两者间分页搬运数据，自己管理自己的记忆。核心抽象是**可编辑的 memory blocks**。

- **存储**：**三级分层**。① **Core memory**：常驻上下文的若干 **memory block**（如 `persona` 角色、`human` 用户画像），有 token 上限，是结构化的"managed context unit"。② **Recall memory**：完整对话历史，自动落盘、可检索。③ **Archival memory**：向量库，存任意长文本/事实，容量近无限、按需检索。是少数**显式分层 + 区分类型（角色/用户/任务）**的方案。
- **提取**：**agent 自驱**。LLM 在推理循环里判断什么值得记，**主动调用 memory 工具**（如 `core_memory_append`、`archival_memory_insert`）写入；不依赖外部抽取管线。新增**睡眠时 agent（sleep-time agents）**：在 idle/无用户输入的回合**异步**整理记忆（合并 archival、重写臃肿的 human block、把近期对话摘要成稳定笔记）。
- **召回**：Core memory **始终注入** prompt（无需检索）；recall/archival 由 agent **按需调用搜索工具**（`archival_memory_search` 等，向量检索）取回并拼入上下文。配合**上下文分页/换出**：上下文将满时把旧消息挤出到 recall（可再搜回），实现"虚拟无限上下文"。
- **冲突处理**：靠 **self-editing tool** 解决——agent 用 `core_memory_replace` / `append` 直接改写 block 内容；过时信息由 agent（或睡眠时 agent）重写覆盖。是"**让 agent 自己编辑**"而非系统规则决策。
- **集成**：自托管 Letta Server / Letta Cloud；REST API + Python/TS SDK + ADE（可视化 agent 调试器）。记忆与 agent 状态持久化在服务端，agent 本身就是有状态的。

## 4. Zep / Graphiti（重点：边失效机制）

**实质**：托管记忆服务，开源核心引擎 **Graphiti** 是一个**双时间轴（bi-temporal）时序知识图谱**——不是向量库，而是把对话沉淀成可随时间演化、可回溯历史真值的实体-关系图。

- **存储**：统一图分**三个子图**。① **Episode**：原始摄入单元（message/text/JSON），保留溯源。② **Semantic Entity**：LLM 抽取的实体(Node)与关系事实(Edge 三元组)，实体带可演化 summary。③ **Community**：实体聚类得到的高层簇。**双时间轴**：每条边四字段 `created_at`（入库）/ `expired_at`（事务层失效）/ `valid_at`（生效起）/ `invalid_at`（生效止）。支持自定义 ontology。
- **提取**：`add_episode()` 摄入后由 **LLM 抽实体与边**，带 reflexion 自检，再做 **entity resolution / 去重**合并跨 episode 的重复指代。**异步**处理。
- **召回**：**混合检索** = 向量余弦 + BM25 全文 + BFS 图遍历，再 rerank（RRF / MMR / 图距离 / cross-encoder）。结果组装成 **Context Block** 注入 prompt，含用户 summary + 相关 facts + 每条 fact 的 `valid_at/invalid_at` 时间元数据。接口 `graph.search` / `thread.get_user_context`。
- **冲突处理（核心）**：新事实**不删除旧边**，而是**置无效保留历史**。新边入库时与已有边做矛盾检测，矛盾的旧边写 `invalid_at`（valid-time 失效）/ `expired_at`（transaction-time 失效）。因而既能查"当前真值"，也能回溯"历史某刻的真值"。论文在 DMR / LongMemEval 上准确率更高、亚秒级延迟。
- **集成**：**Zep Cloud**（托管，多用户/治理/dashboard）；**Graphiti 开源库**（自托管，后端 Neo4j/FalkorDB/Neptune/Kuzu）。SDK：Python/TS/Go。

## 5. A-MEM（Agentic Memory，Zettelkasten 风格）

**实质**：把每条记忆做成一张**结构化卡片（note）**，由 LLM 即时打标，并**自动在卡片间建链接**、新卡片加入后**回溯重写**旧卡片——让记忆网络随经验自组织演化。

- **存储**：粒度=单条交互的 **memory note**，含 7 属性：`content` / 时间戳 / LLM 生成的 `keywords` / `tags` / `contextual description` / `embedding`（对 content+keywords+tags+context 拼接编码）/ `links`。**扁平、无分层、不显式区分记忆类型**，是靠链接互联的笔记网络。
- **提取**：**LLM 写入时**用提示模板从原始内容生成 keywords/tags/context 并编码。写入即结构化（write-time）。
- **召回**：稠密**向量检索**，query 同编码器编码，cosine top-k 注入提示。
- **冲突 / 更新（memory evolution）**：写新 note 时 ① **链接生成**：检索 top-k 相似旧 note，LLM 判断是否真正建链；② **记忆进化**：LLM **回溯更新被链接旧 note 的 context/tags**，更新件替换原件。**无显式过时淘汰 / 容量换页**——靠 LLM 重写消解演变而非删除。
- **集成**：`pip install`，核心类 `AgenticMemorySystem`，方法 `add_note/read/search_agentic/update/delete`。默认 `all-MiniLM-L6-v2` + OpenAI/Ollama + ChromaDB。

## 6. MemoryOS（类操作系统分层记忆）

**实质**：把 OS 内存管理搬到 agent 记忆——**STM/MTM/LPM 三级**，靠 FIFO 与"热度"公式做分页换出与晋升（冷页换出、热页升入长期画像）。EMNLP 2025 Oral。

- **存储**：三级显式分层。**STM**：dialogue page `{Query,Response,Timestamp}` + 上下文链，定长队列（~7 页）。**MTM**：把 page 按主题聚成 segment（语义相似>θ≈0.6），最多 ~200 段，每段带热度。**LPM**：User Persona（静态画像 + 动态知识库 + 演化特质）+ Agent Persona，定长队列（~100 条）。
- **提取**：对话写入 STM；page 按"语义相似 + 关键词 Jaccard"（Fscore）聚进 MTM segment；高热 segment 触发 LLM 抽用户画像/事实写入 LPM。由 Updating 模块分级**异步**驱动。
- **召回**：跨三级检索后拼提示。STM 取全量近期；MTM 两阶段（先 Fscore top-m 段，再语义 top-k page）；LPM 取 top-10 画像/知识。
- **冲突 / 换页淘汰**：**STM→MTM** dialogue-chain FIFO，队满逐出最老页；**MTM 热度** `Heat = α·N_visit + β·L_interaction + γ·R_recency`（访问数/段内对话量/时间衰减）；**MTM→LPM/淘汰**：容量超限按热度淘汰最低分段，热度>τ 的段晋升进 LPM。靠"冷页换出、热页晋升"做更新与遗忘。
- **集成**：`pip install memoryos-pro`，API `add_memory()/get_response()`；提供 **MCP server**（`add_memory/retrieve_memory/get_user_profile`），可接 Claude Desktop/Cursor；兼容 OpenAI/Anthropic/Deepseek/Qwen，ChromaDB + bge-m3，有 Docker / Web playground。

## 8. 对 Replica 的启发建议

针对诉求：**每分身独立记忆、语义(画像/事实)+情景(历史问答)分通道、聊天中沉淀、自进化飞轮**。

### 8.1 记忆该怎么存：分通道 + 原子事实 + 情景留档，两条腿走路

主流共识是**「原子化事实 + 向量召回」**（mem0 / A-MEM 都是），而非整段对话堆 chunk（MemPalace 那种逐字存的弱点已被独立审计点破：无冲突检测、矛盾静默堆积）。建议对应你的两个通道分开建模：

- **语义记忆（画像/事实通道）**：存**原子事实**（一句话级，自然语言或 `(实体,属性,值)` 三元组），每条带 `source_qa_id`（溯源到产生它的那轮问答）、`valid_at`、`confidence`。这是要做 add/update/delete 的部分。可借鉴 **Letta 的 core block** 思路把"高频稳定画像"（岗位/职责/常用术语/性格）固化成一个**常驻 block**，每次对话直接注入，不走检索。
- **情景记忆（历史问答通道）**：**原样留档**每轮 Q&A（不删不改，append-only），向量化做语义召回。这部分**不要做冲突合并**——情景就是历史快照，矛盾是正常的（人会改主意），价值在可回溯。Zep 的 episode 子图、Letta 的 recall 都是这个定位。

> 一句话：**语义记忆做"一条条原子事实 + 冲突合并"，情景记忆做"append-only 问答留档"。两者用同一个 `qa_id` 关联**，飞轮才能从情景反推语义。

### 8.2 提取：异步后处理 + LLM 抽取，别 inline 阻塞

- 用 **mem0 式两阶段**：对话照常回答（实时不阻塞），**异步**用 LLM 从这轮 Q&A 抽语义事实候选。MempPalace 的"纯规则零 LLM"对画像类抽取太弱，不可取。
- 借 **Letta 睡眠时 agent**：分身空闲时跑后台任务，把零散事实**合并、重写、升级**进画像 block——这正好是你"自进化飞轮"的执行载体。

### 8.3 召回：混合检索 + 画像常驻

- **画像 block 常驻注入**（Letta core memory 思路）+ **情景/长尾事实按需向量召回**。若员工关系/项目协作是重点，可像 Zep 那样上**轻量知识图谱**做多跳（"谁负责什么、向谁汇报"），但别一上来就全图谱化（Graphiti 抽取成本与延迟高）。起步用 **向量 + 时间过滤**足够。
- 召回时**带上时间元数据**，让生成模型知道"这是 3 个月前的说法"。

### 8.4 冲突处理：直接抄 mem0 的 ADD/UPDATE/DELETE/NOOP（这是最关键的一条）

这是全场最值得借鉴的机制，**强烈建议语义通道照搬**：

1. 抽出候选事实后，向量检索最相似的已有画像事实；
2. 把「候选 + 相似历史」喂 LLM，让它**四选一**：ADD（全新）/ UPDATE（细化补充）/ DELETE（矛盾，淘汰旧的）/ NOOP（无变化）；
3. **关键改良**：DELETE 不要物理删除，改成 **Zep 式软失效**——给旧事实置 `invalid_at` 并保留，新事实指向被它取代的旧事实 id。这样既保证"当前画像无矛盾"，又留下**画像演变史**（员工成长轨迹），对数字分身和飞轮都是资产。

> 即：**冲突决策用 mem0（LLM 决策），但落库用 Zep（软失效保留历史），取两家之长。**

### 8.5 自进化飞轮：把"沉淀"和"进化"拆成两个异步环

- **沉淀环（每轮对话后）**：Q&A → 情景 append-only → 异步抽语义候选 → mem0 式冲突决策入画像。
- **进化环（定期/空闲）**：睡眠时 agent 扫情景与画像 →（A-MEM evolution 思路）重写聚合零散事实、补全标签、发现高阶模式 → 更新常驻画像 block。两环都异步，不拖慢"和分身聊天"的实时体验。

### 8.6 每分身独立记忆：用作用域隔离

直接采 mem0 的 **`user_id`/`agent_id`/`run_id`** 模型：把**每个分身当一个 `agent_id`**（其记忆按 agent 隔离），**对话方/会话**用 `run_id`。这样天然支持"一个员工一个分身、各自独立长期记忆"，且查询时按 `agent_id` 过滤即可，不必每分身一套独立库。
